# Telegram Bot 处理中消息复刻设计

## 背景

当前 `codex-manager` 的 Telegram Bot 已经能把 Codex 的中间事件流转成一条持续编辑中的“正在处理...”消息，但展示仍偏向事件直出，例如“正在调用 Shell”“Shell 已调用完成”。这类文本能说明系统在工作，却不够接近 Codex 桌面端那种低噪声、连贯、友好的处理中体验。

本次设计的目标不是全量复刻桌面端最终消息结构，而是优先把 **处理中消息** 做得更像 Codex：

- 使用 **单条消息持续编辑**
- 同时展示 **简短过程文字** 与 **阶段摘要**
- 尽量避免刷屏和工具细节泄漏

## 范围

### 本次纳入

- Telegram Bot 处理中消息的结构升级
- 工具事件到阶段摘要的聚合
- 中间过程自然语言与摘要的共同展示
- 已有中断、审批按钮、最终输出替换逻辑的兼容

### 本次不纳入

- 最终结果消息的完整复刻
- Telegram 内部的文件变更可展开视图
- 历史消息回放时重建同款摘要
- 撤销、回滚、diff 浏览等桌面端能力
- app-server 协议的大规模调整

## 设计目标

1. 让用户看到一条稳定成长的处理中消息，而不是日志流。
2. 把原始工具事件压缩成更自然的阶段进展，例如“已探索 2 个文件，ran 2 commands”。
3. 保留少量思考/过程文字，维持“有人在处理任务”的体感。
4. 改动集中在 Telegram bot 层，降低对现有协议和桌面端代码的影响。

## 方案对比

### 方案 A：事件直出增强

保留现有 `StreamPreviewState` 的字符串拼接方式，只把文案改得更自然。

优点：

- 改动最小
- 上线最快

缺点：

- 本质仍是日志流
- 很难做出“已探索 / 已编辑 / ran commands”这种阶段感
- 与目标体验仍有明显差距

### 方案 B：处理中消息聚合器

把 `StreamPreviewState` 升级为“处理中消息聚合器”，将事件拆成结构化状态，再统一渲染为一条 Telegram 消息。

优点：

- 最接近目标体验
- 改动主要集中在 Telegram bot 层
- 可逐步扩展更多摘要类型

缺点：

- 需要补一层事件归纳逻辑
- 部分摘要需要从现有 item 信息做保守推断

### 方案 C：结果导向型摘要

弱化过程文字，主要显示“已探索 / 已编辑 / ran commands”这类短摘要。

优点：

- 展示更克制
- 文案一致性更高

缺点：

- 少了过程文字，临场感不足
- 与参考截图相比不够像

### 结论

采用 **方案 B**。

## 处理中消息结构

持续编辑中的 Telegram 消息固定由三部分组成。

### 1. 顶部状态区

职责：

- 稳定告诉用户任务仍在进行
- 提供简短耗时信息

示例：

- `正在处理...`
- `正在处理...`
- `已处理 38s`

要求：

- 文本稳定，避免每次刷新都大改首行
- 审批中和普通处理中共用同一骨架

### 2. 过程文字区

职责：

- 展示最近的自然语言过程说明
- 维持 Codex 正在思考和执行的体感

规则：

- 优先保留完整句子或完整段落
- 新的 `Message` 到来时，替换掉仅由 `Delta` 形成的临时内容
- 最多保留最近 `1-3` 段，避免消息无限增长
- 不回显工具参数、命令全文、敏感输出

### 3. 阶段摘要区

职责：

- 用低噪声方式告诉用户当前推进到了哪一步
- 弱化原始工具事件

示例：

- `已探索 2 个文件，ran 2 commands`
- `已编辑 2 个文件`
- `已探索 1 次搜索`

规则：

- 摘要单独成区，不和过程文字揉成一段
- 优先覆盖与合并，不无限堆积
- 最多保留最近 `2-3` 条摘要

## 组件设计

### `StreamPreviewState` 升级方向

当前 `StreamPreviewState` 更像是“文本累加器”。本次把它升级为状态聚合器，内部至少维护：

- `header_state`：处理中状态和耗时显示
- `agent_messages`：最近完整过程文字
- `pending_delta_text`：尚未被完整消息替换的临时文字
- `tool_window`：最近一段时间内的工具事件窗口
- `summary_lines`：已生成的阶段摘要
- `stats`：探索/编辑/命令/搜索等累计计数

### 建议新增的内部职责函数

- `ingest_progress(progress)`：消费单个进度事件
- `merge_agent_text(...)`：处理 `Delta` / `Message` 合并
- `record_tool_event(...)`：记录工具开始/完成
- `flush_summary_if_needed(...)`：在合适时机产出摘要
- `render_processing_message(...)`：生成最终编辑到 Telegram 的文本

这会让“事件消费”和“最终文案渲染”分离，后续更好维护。

## 数据流

### 输入

输入仍然来自：

- `spawn_stream_preview(...)`
- `progress_rx: mpsc::UnboundedReceiver<app_server::TurnProgress>`

### 中间处理

1. 收到 `Delta` 或 `Message` 时，更新过程文字区。
2. 收到 `ToolStarted` / `ToolCompleted` 时，更新工具窗口与统计信息。
3. 达到刷新条件后，由聚合器统一生成新的处理中消息文本。
4. 通过 `edit_text_with_reply_markup(...)` 更新同一条 Telegram 消息。

### 输出

输出仍是一条 Telegram 消息的持续编辑，不改变现有“停止本轮”“审批按钮”“最终结果替换”的外层交互流程。

## 阶段摘要归纳规则

第一版只实现三类摘要，避免范围失控。

### 探索类摘要

来源：

- `Shell` 中的 `read / listFiles / search`
- `Web 搜索`
- `查看图片`
- 只读型 MCP 工具（保守纳入）

输出方向：

- `已探索 N 个文件`
- `已探索 N 次搜索`
- `已探索 N 个文件，ran N commands`

### 编辑类摘要

来源：

- `fileChange`

输出方向：

- 优先：`已编辑 N 个文件`
- 退化：`已编辑 1 处变更`

### 执行类摘要

来源：

- `commandExecution`

输出方向：

- `ran N commands`
- 第一版保留中英混合，以贴近参考体验

## 摘要刷新策略

摘要不应随每个事件都刷新。第一版采用保守刷新策略：

1. 工具完成后优先刷新，而不是工具刚启动就刷新。
2. 同类型事件在短时间内合并。
3. 新摘要优先覆盖旧摘要的同类表述。
4. 同时最多显示最近 `2-3` 条摘要。

这样可以减少 Telegram 的频繁改写和用户感知上的噪声。

## 失败与降级策略

如果现有事件不足以稳定给出细节，不做激进推断。

### 文件数拿不到

降级为：

- `已编辑 1 处变更`
- 或不生成编辑条目

### 命令动作解析拿不到

降级为：

- `ran N commands`

### 搜索详情拿不到

降级为：

- `已探索 1 次搜索`

### 过程文字不足

保留顶部状态区和摘要区，允许处理中消息暂时只显示摘要。

## 错误处理

### Telegram 编辑失败

保持现有策略：

- `message is not modified` 视为成功
- 其他错误记录日志并停止当前预览任务

### 事件顺序不稳定

聚合器应按“最终一致”思路工作：

- `Message` 可覆盖临时 `Delta`
- 同类摘要以后到的结果为准

### 审批流

审批相关文案继续沿用现有按钮逻辑，但在渲染时应保证：

- 审批提示能和普通处理中消息共存
- 审批解决后恢复普通处理中结构

## 测试策略

本次测试应集中在 `telegram-codex-bot.rs` 的纯逻辑层。

### 需要新增或调整的单元测试

1. `Delta` 被完整 `Message` 替换，而不是重复累加
2. 多条过程文字只保留最近有限段落
3. 工具事件被正确归入探索/编辑/执行
4. 多次命令执行被合并成 `ran N commands`
5. `fileChange` 被汇总成 `已编辑 N 个文件` 或降级文案
6. 摘要不会无限增长
7. 审批事件不会破坏处理中消息骨架

### 验证标准

- 处理中消息从“事件列表”变成“结构化过程展示”
- 无命令参数、无工具原始输出泄漏
- 同一条 Telegram 消息持续编辑
- 现有停止与审批流程不回归

## 实施建议

推荐分两步：

1. 先完成聚合器与摘要渲染，保留现有事件来源
2. 如果某些摘要无法稳定生成，再补最小协议增强

这样能先把体验显著拉近目标，再决定是否值得继续深挖协议细节。

## 成功标准

当用户在 Telegram 中发起任务时，应能看到：

- 一条持续编辑中的处理中消息
- 最近的简短自然语言过程说明
- 低噪声的阶段摘要，如“已探索 / 已编辑 / ran commands”
- 明显比当前“正在调用 X / X 已完成”的事件直出更接近 Codex 桌面端体验
